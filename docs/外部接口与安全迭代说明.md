# 外部接口与安全迭代说明

本文档说明本轮新增的禁忌管理、营养素推荐排序、病例识别兜底和外部接口能力。所有改动遵循既有安全原则：产品目录、禁忌规则、医生规则和人工审核优先，系统不得因为 RAG 或外部接口绕过安全边界。

## 1. 产品禁忌与慎用提示

新增数据文件：

- `backend/app/data/product_safety_matrix.json`

该文件由药企提供的《产品服用禁忌汇总》整理生成，原始 DOCX 不进入 Git。运行时会把安全矩阵覆盖到产品目录中，补充以下信息：

- `contraindications`：明确禁用或禁忌场景。
- `cautions`：慎用场景，可推荐但必须提示顾问或医生确认。
- `interaction_warnings`：药物、手术、特殊人群等交互风险。
- `exclusion_rules`：用于自动排除的结构化规则。

处理原则：

- 出现明确“禁用、禁忌、避免使用”等表达时，作为自动排除规则。
- 出现“慎用、需监测、需确认”等表达时，不直接排除，但进入推荐草案和报告提示。
- 产品目录中不存在或未启用的条目不会被新增为可推荐产品。

## 2. 产品标签矩阵排序优化

本轮对产品标签矩阵的排序权重做了更细分的处理：

- 症状命中：优先级最高，用于把与当前主诉直接相关的营养素排在前面。
- 既往病史/问题命中：用于识别甲状腺、血糖、血脂、胃肠、炎症等明确问题。
- 健康目标命中：用于配合“控糖、睡眠、免疫、代谢”等目标。
- 生活方式命中：作为辅助因素，避免只因泛化描述把弱相关产品提前。

推荐草案中的 `evidence_details` 会显示类似“产品标签命中：症状”的证据标签，方便内部审核。客户可见报告仍只展示经过整理后的推荐理由，不展示内部打分。

## 3. 病例识别兜底

为减少新增医院模板导致的漏识别，本轮增加两类防线：

1. 单位归一化增强
   - 支持 `10 9/L`、`10*9/L`、`10×9/L`、`10^9 / L` 等形式统一为 `10^9/L`。
   - 支持 `μmol/L`、`µmol/L`、部分 OCR 误识别字符统一为 `umol/L`。

2. 疑似未识别指标池
   - 对“像检验指标但未被字典识别”的行，加入人工校对提示。
   - 示例：`疑似未识别指标：载脂蛋白E 1.82 g/L ↑ 0.20-1.20`
   - 该机制不会把未知指标强行归一化，也不会直接参与推荐，只用于提醒人工审核。

## 4. 外部接口

外部接口统一挂载在 `/api/v1` 下，使用 Bearer token。Bearer token 复用现有医生账号登录后的服务端 session，不新增明文密码或 JWT 私钥体系。

### 4.1 登录获取 Token

`POST /api/v1/auth/token`

请求：

```json
{
  "username": "doctor-a",
  "password": "secret123"
}
```

响应：

```json
{
  "access_token": "sess_xxx",
  "token_type": "bearer",
  "expires_in_days": 14,
  "doctor_id": "doctor_xxx",
  "display_name": "医生A"
}
```

后续接口需携带：

```http
Authorization: Bearer sess_xxx
```

### 4.2 创建病例

`POST /api/v1/cases`

请求：

```json
{
  "customer_name": "客户姓名",
  "consultant_id": "顾问姓名",
  "notes": "可选备注",
  "analysis_mode": "llm_primary"
}
```

说明：

- 外部接口创建的病例默认进入医生私有工作区。
- 只有创建该病例的医生 token 可以继续上传附件、生成推荐或获取报告。

### 4.3 上传病例或问卷附件

`POST /api/v1/cases/{case_id}/attachments`

表单字段：

- `files`：一个或多个附件。
- `attachment_type`：`case` 或 `questionnaire`。

返回：

```json
{
  "case_id": "case_xxx",
  "status": "parsing_completed",
  "results": [
    {
      "file_id": "file_xxx",
      "filename": "labs.txt",
      "attachment_type": "case",
      "status": "parsed",
      "lab_item_count": 2,
      "parse_warnings": []
    }
  ]
}
```

说明：

- `case` 附件会进入 OCR/文本抽取和指标识别流程。
- `questionnaire` 附件会进入问卷识别流程。
- 若出现疑似未识别指标，会写入 `parse_warnings`，供人工校对。

### 4.4 生成营养素推荐 JSON

`POST /api/v1/cases/{case_id}/nutrition-recommendations`

返回：

```json
{
  "case_id": "case_xxx",
  "draft_id": "draft_xxx",
  "status": "pending_review",
  "manual_review_required": true,
  "confidence": 0.78,
  "recommendations": [
    {
      "sku_id": "sku_xxx",
      "name": "产品名称",
      "category": "营养素类别",
      "dosage": "每日 1 粒，随餐使用。",
      "reason": "推荐理由",
      "warnings": ["慎用或交互提示"]
    }
  ],
  "contraindications": [],
  "missing_info": []
}
```

说明：

- 该接口只生成草案，不自动发布客户报告。
- 草案仍需人工审核。
- 明确禁用产品会被排除，慎用产品可保留但必须带提示。

### 4.5 获取最新营养素推荐

`GET /api/v1/cases/{case_id}/nutrition-recommendations/latest`

用于外部系统拉取当前病例最近一次推荐草案。

### 4.6 获取报告下载地址

`GET /api/v1/drafts/{draft_id}/report-download`

说明：

- 仅已审核发布的草案可返回下载地址。
- 未审核草案返回 `409 Report is not approved yet`。

响应：

```json
{
  "draft_id": "draft_xxx",
  "filename": "客户报告.pdf",
  "download_url": "http://localhost:8000/api/v1/drafts/draft_xxx/report.pdf"
}
```

PDF 下载：

`GET /api/v1/drafts/{draft_id}/report.pdf`

同样需要 Bearer token，且只允许病例所属医生访问。
