# 新成员协作与源码交付指南

## 推荐交付方式

当前项目仍处于开发阶段，推荐采用“私有 Git 仓库交付源码 + Docker Compose 统一启动环境 + 医学资料分级管理”的方式协作。

- Git 负责源码交付、多人修改、版本历史、分支合并和回退。
- Docker Compose 负责统一启动前端、后端和本地运行目录，减少环境差异。
- 医学原始资料、真实病例、运行数据库和密钥不进入 Git 仓库。

不建议只发 Docker 镜像给参与开发的新成员。镜像适合体验、测试或交付部署，但不适合作为源码协作方式。

## 仓库内容边界

可以进入私有 Git 仓库：

- 前端代码、后端代码、Docker 配置、脚本、README 和项目文档。
- `.env.example` 等配置模板，不包含真实密钥。
- `backend/app/data/` 下已经整理好的 JSON 规则、产品目录、指标字典和结构化知识数据。
- 从书籍或资料中提取、改写、结构化后的规则，例如营养素推荐、禁忌、指标解释、生活方式建议和问卷映射。
- 少量脱敏测试样例，前提是不包含客户身份信息和完整原始病例。

不要进入 Git 仓库：

- `.env`、`.runtime/`、SQLite 运行数据库、上传文件和导出报告。
- 真实体检报告、客户问卷、截图、病例资料和任何可识别个人身份的信息。
- 原始 PDF、Word、Excel、扫描图片、完整电子书和大段未改写教材原文。
- 本机资料目录，例如 `体检报告/`、`功能医学相关资料/`、`AI学习功能医学相关资料/`。

## 新成员启动流程

负责人先在 GitHub Private、Gitee 私有仓库或 GitLab 中创建私有仓库，并邀请新成员加入。

新成员本地启动：

```bash
git clone 私有仓库地址
cd 项目目录
cp .env.example .env
docker compose up --build
```

如果需要外部知识资料包，在项目根目录创建 `knowledge/`，再把已授权的资料放入该目录。该目录默认不进入 Git，也不会被打包进 Docker 构建上下文。

默认访问地址：

- 前端工作台：`http://localhost:3000`
- 后端接口：`http://localhost:8000`
- 健康检查：`http://localhost:8000/health`

## 日常协作流程

- `main` 分支只保留稳定代码，不直接用于日常开发。
- 每个功能或修复从 `main` 新建分支，例如 `feature/nutrient-rules`、`fix/pdf-export`。
- 每次完成一小块可验证工作后提交 commit，说明改了什么和为什么改。
- 合并前至少确认 Docker Compose 能启动，核心流程能跑通。
- 团队人数增加后，使用 Pull Request / Merge Request 做合并前检查。

建议分工：

- 负责人把控业务方向、医学规则、营养素推荐逻辑和报告最终效果。
- 新成员初期优先处理前端界面、PDF 样式、文档、测试用例和低风险 bug。
- 涉及病例权限、规则命中、营养素推荐和知识库结构的改动，先讨论方案再开发。

## 资料交付建议

结构化 JSON/CSV 可以随源码进入 Git，但必须先完成脱敏和改写。原始医学资料建议通过企业网盘、加密网盘、内网共享或单独资料包交付，并按成员权限控制。

如果新成员只需要开发界面和通用流程，优先给脱敏样例和结构化知识数据；如果需要调试知识库导入，再单独提供最小化、已授权的资料包。
